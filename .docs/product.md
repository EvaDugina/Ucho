## project

- **Название:** Psycho
- **Тип:** Telegram-бот с локальным LLM, граф знаний в Obsidian
- **Краткое описание:** Личный бот-собеседник, который через серию вопросов извлекает из ответов пользователя принципы, ценности и убеждения, складывает их в граф концептов в Obsidian-vault и подсвечивает противоречия.

---

## purpose

Помочь владельцу построить и поддерживать живую карту собственного внутреннего мира: что я считаю правильным, что красивым, во что верю, как отношусь к работе/смерти/национальности и т.д. Карта строится не одним приёмом, а в режиме непрерывного диалога — бот задаёт точные вопросы, разбирает ответы на «атомы» (концепты), связывает их между собой и сам ловит моменты, когда новые высказывания противоречат старым.

Ценность — не сам граф, а **давление на собственные взгляды**: систематические вопросы заставляют формулировать то, что обычно не формулируется, а ловля противоречий показывает места, где мысль сама с собой не сходится.

## context

- **Домен:** личный самоанализ + ведение Obsidian-vault как «второго мозга».
- **Бизнес-окружение:** проект не клиентский. Один пользователь, своя машина (Ryzen 5 5600 / 32 GB / RTX 3060 12 GB), Obsidian-vault в YandexDisk-папке для синка между устройствами.
- **In scope:** диалог через Telegram, локальный LLM-разбор ответов, запись концептов/связей/конфликтов в vault, Obsidian-нативный формат (callouts, aliases, block refs, per-domain MOC), 10 предметных доменов.
- **Out of scope:** многопользовательский режим, веб-интерфейс, монетизация, экспорт в другие форматы, голосовые сообщения, картинки, любая телеметрия наружу.
- **Допущения:** владелец читает результат в Obsidian, а не в боте; правки в Obsidian руками — допускаются и не должны теряться при последующих автозаписях бота.
- **Риски:**
  - Vault лежит в YandexDisk-папке — фоновый синк может конкурировать с записями бота (закрыто atomic writes + manifest drift detection).
  - Локальная LLM на GPU 12 GB — Qwen 2.5 14B иногда галлюцинирует структуру JSON; закрывается строгой валидацией и фолбэками с логом.
  - Один пользователь = один источник правды; бэкап графа сейчас опирается на git внутри vault + YandexDisk-history.

## style

- **Персона:** бот говорит голосом **Иуды из Кариота** (по повести Андреева) — двуликий, вкрадчиво-колкий собеседник, который любит человека и потому не верит его красивым словам. Не «коуч», не «дневник»: подсовывает зеркало, нащупывает зазор между тем, каким человек себя считает, и тем, каков он есть. Без эмодзи в ответах LLM, без воды, без морали и утешения.
- **UI-стиль:** Telegram нативный, минимум разметки. Заголовок вопроса в формате `Q<N> · [mode ·] _domain_`, тело — `<code>…</code>` (для long-press копирования). Спиннер 🎰/🎲/🎯 на время ожидания LLM.
- **Тон коммуникации:** на «ты», провокативно-интимный. Бот поддевает, сомневается вслух, бьёт по самооправданиям — но не выдумывает противоречий там, где их нет.
- **Терминология:** «концепт» (узел графа), «тема» (предметная область = поле `domain`; на графе — цвет узла + тег-хаб `#ТЕМА` заглавными, напр. `#AESTHETICS`), «сквозная тема» (русский тег-мотив, связывает концепты через разные домены, напр. `#присутствие`), «связь» (`supports/contradicts/derived_from/related`), «алиас» (альтернативная формулировка), «raw» (сырой Q&A-лог дня), «MOC» (оглавление темы; на графе скрыт — его роль хаба берёт тег).

## features

### Core

- **10 тем вопросов:** ethics, aesthetics, politics, everyday, relationships, identity, mortality, nationality, knowledge, work. Выбор темы через inline-кнопки или аргумент команды; «на выбор бота» = равномерный random.
- **Иерархия вопросов в probe-режиме:** один главный вопрос → ответ → до одного поясняющего → ответ → автоматический закрывающий комментарий-сводка. После сводки сессия закрывается. Лимит `MAX_CLARIFIERS=1` в коде.
- **Capture-first разделение труда:** бот (Qwen 14B) только *захватывает* — создаёт **черновые** концепты `status: draft` без связей и без конфликтов. Выверенный граф (промоушн `draft → stable`, связи, реальные противоречия, дедуп, profile, MOC, теги, русские имена) строит сильная модель (Claude) раз в неделю через скилл `weekly-review`. Причина: Qwen слаба в графовом рассуждении — плодит дубли и ложные конфликты.
- **LLM только анализирует — запись в БД делает код:** в режиме `process` модель возвращает лишь `observations` (что человек выдал о себе: `domain/type/name/summary/quote`) и следующий вопрос. Она НЕ придумывает slug, НЕ решает create-vs-update, НЕ пишет raw. Всё это детерминированно в коде: raw пишется дословно, slug выводится из имени (транслит), дубли ловит дедуп. Это снижает нагрузку на локальную модель и повышает надёжность JSON (важно под стилизованной персоной).
- **Свободные заметки (`/ucho`):** verbatim в `notes/<дата>.md` + прогон через тот же `process` — черновые концепты попадают в граф (заметка работает как ответ без заданного вопроса).
- **Дедупликация (live, лёгкая, в коде):** для каждого `observation` код прогоняет `resolve_slug` (по имени → по выведенному slug → aliases) + Jaccard-overlap на биграммах summary (0.7) против существующих в домене. Совпадение → дописать evidence + alias, не создавать дубль. Глубокий дедуп/слияние — у Claude еженедельно.
- **Ревью базы (`/review`):** диалог в свободной форме о том, что лежит в графе; бот может предложить добавить недостающие концепты, пользователь подтверждает `да/нет`.
- **Постоянство:** активная сессия, сквозной счётчик Q-номеров и недоработанный ответ (`pending_answer`) переживают рестарт контейнера (`_session.json`, `_state.json`, atomic writes). Офлайн-ответы доезжают из очереди Telegram при следующем старте polling.
- **Self-check при старте:** механический (без LLM) прогон — MOC rebuild по всем доменам, валидация связей, отчёт о дублях/сиротах в `.psycho/startup-check.md`.
- **Возврат к старым вопросам:** `/history` показывает все Q&A; `/requestion N` задаёт заново.
- **Ответ на выбранный вопрос (reply / `/answer N`):** можно ответить не только текущему, но и любому недавнему вопросу — Telegram-reply на сообщение-вопрос или `/answer N <текст>`. Решает сценарий «бот спросил → я отвлёкся на команду → вопрос остался без ответа»: такой вопрос всё ещё отвечаем. Первый ответ на неотвеченный пишется под его исходным Q-номером; повторный ответ на уже отвеченный — как Q-повтор (новый блок, старый ответ цел). Карта `message_id→вопрос` (последние 50, переживает рестарт) — единственный след неотвеченного вопроса, в `raw/` его ещё нет.
- **Ежедневный пинок:** APScheduler шлёт главный вопрос в `DAILY_HOUR`, если сессии нет.

### Secondary

- **Obsidian-native рендеринг:** callouts `[!summary]/[!quote]/[!question]/[!contradiction]/[!source]`, aliases во frontmatter, block refs `[[raw/<date>#^Q<n>]]` (точная навигация в raw).
- **Наглядный русский граф (строит weekly-review):** у выверенных (`stable`) концептов имя файла = `slug` = русский заголовок — узлы графа подписаны по-русски (не транслитом). Все ссылки ведут по русскому имени; старый ascii-slug уходит в `aliases`. Узлы окрашены по домену (10 цветовых групп), узлы-теги серые. Бот не перезатрёт такой концепт — `safe_slug` принимает только ASCII, русский slug служит естественным write-barrier.
- **Теги как второе измерение графа:** на концепте — доменный тег `#ТЕМА` (= имя папки заглавными, хаб домена) + 1–3 **сквозные** русские темы из закрытого реестра `_tags.md`. Сквозные темы связывают концепты ЧЕРЕЗ разные домены (то, чего не дают папки и ссылки). Графовые настройки (фильтр только `concepts`, скрытие MOC-файлов, цвета, серые теги) лежат в `.obsidian/graph.json`; для нового пользователя бот сидит их из шаблона `bot/assets/graph.json`.
- **Per-domain MOC:** `concepts/<domain>/<DOMAIN>.md` — оглавление темы, концепты сгруппированы по type, пересобирается после каждого изменения. На графе **скрыт** фильтром (`-file:<DOMAIN>`) — роль хаба домена берёт тег `#ТЕМА`.
- **Защита ручных правок:** если файл концепта правился в Obsidian (или YandexDisk pull-нул внешнюю правку), бот **не перезаписывает** — записывает warn в `.psycho/log.md` и сохраняет правки пользователя.
- **Операционный лог:** `.psycho/log.md` append-only, фиксирует drift-skip, dedup, sanitize, LLM-фолбэки.
- **Спиннер:** анимированный dice + текстовый placeholder во время ожидания LLM. Placeholder удаляется по готовности, dice остаётся как маркер «здесь думал».

### Out of scope

- Открытый/публичный multi-user, регистрация, MFA, 152-ФЗ-режим — нет. Multi-user есть, но **только для нескольких доверенных** по whitelist (владелец добавляет вручную); у каждого изолированная база. Открытый доступ и связанный с ним hardening — MVP B.
- Веб-интерфейс / мобильное приложение — клиент только Telegram, чтение результата только в Obsidian.
- Облачный LLM (OpenAI/Anthropic) — приватность приоритетна, всё локально через Ollama.
- Голосовые сообщения, картинки, файлы — только текст.
- Экспорт графа в другие форматы (Notion, Roam, JSON-API) — Obsidian-vault считается source of truth.
- Бэкапы по cron, мониторинг, дашборды — это обвязка MVP A, на PoC B не делаем.
- Миграция исторических концептов 4→10 доменов — есть отдельный скрипт `scripts/migrate_domains.py`, но он не запущен; до запуска старые концепты в legacy-доменах продолжают читаться.

## quality

**Нефункциональные требования:**

- **Приватность:** ни один токен ответа пользователя не покидает локальную машину. Все LLM-вызовы идут через Ollama по `OPENAI_BASE_URL=http://ollama:11434/v1`.
- **Сохранность данных:** ручные правки в Obsidian не должны теряться при последующей автозаписи. Закрыто manifest+mtime drift detection.
- **Целостность под падением:** kill контейнера в момент записи не должен оставить битый JSON / полу-записанный концепт. Закрыто atomic writes (tmp + os.replace).
- **Откат на ошибку:** падение посреди `_apply_processed` должно откатывать vault к pre-state. Закрыто `git_wrap` транзакцией (pre/post commit, на исключение `git reset --hard`).
- **Latency:** на RTX 3060 + Qwen 14B главный вопрос генерируется за 5–20 сек, обработка ответа 10–30 сек. На CPU (Ryzen 5 5600) с Qwen 7B — 5–15 сек на вопрос, терпимо.

**Метрики успеха (для PoC B):**

- За неделю владелец отвечает ≥10 вопросов без падения бота.
- Граф в Obsidian-Graph-View визуально читается: видны кластеры по доменам, цветные связи между концептами.
- Ни один автоматический коммит бота в vault не разрушил ручную правку.
- Минимум один conflict-callout записан и помог осознать противоречие в собственной позиции.

## rules

### Hard

- LLM-вызовы только локально. Никаких облачных провайдеров.
- Все команды и подсказки видны только владельцу.
- Vault не теряет ручных правок: drift → skip + log, не overwrite.

### Firm

- Концепты — атомарные. Один концепт = одна идея, не лонгрид.
- **Slug выводит код, не LLM.** Черновик: ascii-slug из имени через `slugify` (транслит). Выверенный (`stable`, weekly-review): имя файла = slug = русский заголовок (наглядный граф), старый ascii-slug — в `aliases` (якорь дедупа + back-compat ссылок).
- Парсер обратимо понимает старый формат (H2-секции) — на случай старых концептов и ручного редактирования.

### Soft

- В Telegram-сообщениях минимум разметки. Заголовок коротко, тело — `<code>`.
- LLM-фолбэки (неизвестный домен, кривой type/status) — логируем, не падаем.

## accept

PoC B считается пройденным, когда:

- Бот запускается одной командой (`docker compose up -d`) и не падает в течение недели.
- Все 10 доменов реально работают: `/ask <domain>` для каждого даёт осмысленный вопрос, ответ корректно разбирается LLM, концепты появляются в `concepts/<domain>/`.
- Открытие MOC-ноды `<ТЕМА>.md` в Obsidian показывает все концепты темы, сгруппированные по type.
- Open Graph View с фильтром `path:concepts/` визуально читается — узлы и связи на месте.
- Конфликт-callout (`[!contradiction]`) реально записан хотя бы раз и виден в Reader-режиме Obsidian.
- `.psycho/log.md` существует, наполняется и не пустой.
- При ручной правке `.md` в Obsidian → следующий ответ не перезатёр правку.

## notes

### Продуктовые решения

- **2026-05-21:** **ответ на конкретный вопрос через reply / `/answer N`.** Закрывает сценарий «бот задал вопрос → пользователь выполнил команду → вопрос остался без ответа, но терять его не хочется». Теперь reply на сообщение-вопрос (или `/answer N <текст>`) открывает probe-сессию, заякоренную на этот вопрос. Решения по развилкам: `/answer` — только инлайн (`/answer N <текст>`, без текста — подсказка); поясняющие вопросы (клариферы) отвечаемы через reply наравне с главными; смена вопроса молча закрывает текущую сессию; первый ответ на неотвеченный — под исходным Q-номером, повторный на отвеченный — Q-повтор; карта вопросов — последние 50, вытесненный вопрос даёт мягкий отказ. Команда `/answer` ранее (см. ниже) была убрана как дубль `/requestion` — здесь возвращается с **другим** смыслом: не «задать заново», а «ответить на заданное».

Промпты (`system/review/summarize` + `seeds`) переписаны в этот голос: двуликий провокатор, подсовывает зеркало, бьёт по самооправданиям. Механика не изменилась — характер живёт в формулировках вопросов/комментариев, JSON-контракт строгий.

Причина: Qwen в реальном диалоге выдумывала ложные противоречия (пользователь их отвергал) и плодила дубли. Live-фича, которую слабая модель делает плохо, заменена на честную еженедельную от сильной. Embedding-модель для дедупа/поиска (`nomic-embed-text` в Ollama) сознательно отложена до MVP A.
