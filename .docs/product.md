## project

- **Название:** Psycho
- **Тип:** Telegram-бот с OpenAI-compatible live-LLM (OpenRouter preferred, AITunnel fallback), граф мировоззрения в Obsidian
- **Краткое описание:** Личный бот-собеседник, который через серию вопросов сохраняет ответы, извлекает из них черновые worldview-атомы и раскладывает их по четырём уровням мировоззрения: `01_Мироощущение`, `02_Миропонимание`, `03_Ценностно-нормативная подсистема`, `04_Практический уровень`. Связи и реальные противоречия собираются отдельным ручным strong-pass, не live-ботом.

---

## purpose

Помочь владельцу построить и поддерживать живую карту собственного мировоззрения: как мир ощущается, как он рационально понимается, какие ценности и нормы считаются значимыми, и как всё это становится жизненной позицией/действием. Карта строится не одним приёмом, а в режиме непрерывного диалога: live-бот программно выбирает область → категорию → тему, задаёт вопрос, сохраняет ответ и создаёт черновые атомы, а связи между ними и настоящие противоречия собирает сильная модель вручную через `reconcista`.

Ценность — не сам граф, а **давление на собственные взгляды**: систематические вопросы заставляют формулировать то, что обычно не формулируется, а ловля противоречий показывает места, где мысль сама с собой не сходится.

## context

- **Домен:** личный самоанализ + ведение Obsidian-vault как «второго мозга».
- **Бизнес-окружение:** проект не клиентский. Один владелец + несколько доверенных пользователей, Obsidian-vault живёт на сервере/хосте; перенос и синхронизация между окружениями идут только через git.
- **In scope:** диалог через Telegram, live-LLM разбор ответов через OpenRouter/AITunnel, обязательная запись session-log/notes до LLM, draft-атомы мировоззрения в vault, Obsidian-нативный формат (callouts, aliases, block refs, path-wikilinks, area-MOC), четыре уровня мировоззрения 01-04. Связи/конфликты — отдельный ручной проход `reconcista`.
- **Out of scope:** публичный многопользовательский сервис, веб-интерфейс, монетизация, экспорт в другие форматы, голосовые сообщения, картинки, любая телеметрия наружу.
- **Допущения:** владелец читает результат в Obsidian, а не в боте; правки в Obsidian руками — допускаются и не должны теряться при последующих автозаписях бота.
- **Риски:**
  - Серверный vault может меняться внешним `git pull` или ручными правками в Obsidian; конкуренция с записями бота закрыта atomic writes + manifest drift detection.
  - Внешний LLM provider может быть нестабилен по лимитам/доступности; закрывается строгой JSON-валидацией, логом и fallback-маршрутом.
  - Один пользователь = один источник правды; бэкап графа сейчас опирается на git внутри vault и серверный git remote.

## style

- **Персона:** бот говорит голосом **Иуды из Кариота** (по повести Андреева) — двуликий, вкрадчиво-колкий собеседник, который любит человека и потому не верит его красивым словам. Не «коуч», не «дневник»: подсовывает зеркало, нащупывает зазор между тем, каким человек себя считает, и тем, каков он есть. Без эмодзи в ответах LLM, без воды, без морали и утешения.
- **UI-стиль:** Telegram нативный, минимум разметки. Заголовок вопроса в формате `Q<N> · [mode ·] _область / категория / тема_`, тело — `<code>…</code>` (для long-press копирования). Индикатор «Думаю» (один стикер 🎰 + текст) — только при генерации вопроса (`/ask`) и портрета (`/about`); реакции в диалоге идут молча.
- **Тон коммуникации:** на «ты», провокативно-интимный. Бот поддевает, сомневается вслух, бьёт по самооправданиям — но не выдумывает противоречий там, где их нет.
- **Терминология:** «worldview-атом» (узел графа мировоззрения), «область» (один из уровней 01-04), «категория» (структурный срез внутри области), «тема» (конкретный фокус вопроса), «связь» (`supports/contradicts/derived_from/related/influences/manifests_as`), «алиас» (альтернативная формулировка), «raw» (сырой Q&A-лог дня), «MOC» (оглавление области).

## features

### Core

- **Таксономия вопросов 01-04:** вопрос выбирается программно: случайная область (`01_Мироощущение`, `02_Миропонимание`, `03_Ценностно-нормативная подсистема`, `04_Практический уровень`) → случайная категория внутри области → случайная тема внутри категории. `/ask <свободный текст>` включает targeted-режим: код мапит запрос к ближайшей категории/теме или создаёт одноразовый `custom_theme`, не меняя канон.
- **Модель диалога (открытая сессия-обсуждение):** главный вопрос **открывает сессию**; на каждый ответ бот даёт **реакцию-укол от первого лица** (НЕ вопрос — уточняющих вопросов бот не задаёт) и ждёт следующего сообщения. Сессия закрывается **только** когда задан новый главный вопрос (`/ask`/дневной/`/echo`) или выполнена команда; исключения — `/pebble`, `/regen`, `/like`, `/remask`, предупреждение `/leta`. Всегда ≤1 открытой сессии. Старую сессию можно продолжить, ответив (reply) на любое её сообщение: поиск идёт по `00_raw/sessions`.
- **Многоликий Иуда (настроение → лицо):** на каждый ответ владельца бот сперва дешёвым LLM-вызовом классифицирует настроение человека по шкале **PAD** — валентность / энергия / **доминирование** (контроль ↔ бессилие) + направленность + фоновая эмоция. Код считает вектор по сессии и выбирает контрастное «лицо». У каждого пользователя отдельно есть curated `05_Общее/mask_frequencies.json` — его пишет только depersonalization, а бот ведёт `05_Общее/mask_frequencies_draft.json` после каждого ответа и после первого `/like` по реплике. События и feedback настроения живут в `01_Мироощущение/mood/`.
- **Анализ полного канона 01_Мироощущение:** для владельца при `ANALYSIS_ENABLED=true` бот отдельным API-вызовом AITunnel/OpenRouter ищет draft-кандидаты по 6 категориям и 103 темам `01_Мироощущение` (`emotions`, `mood_background`, `world_tone`, `beauty_ugliness`, `body_and_energy`, `existential_feeling`). Локальные сигналы PAD/VAD/EmoLex/Dostoevsky служат только подсказками; запись остаётся capture-first через обычные draft-атомы и evidence-цитаты, без локальных ML-моделей и без вывода отчёта в Telegram.
- **Анализ полного канона 02_Миропонимание:** для владельца при `ANALYSIS_ENABLED=true` бот отдельным API-вызовом AITunnel/OpenRouter ищет draft-кандидаты по 6 категориям и 82 темам `02_Миропонимание` (`knowledge`, `beliefs`, `principles`, `causality`, `self_world_model`, `uncertainty`). Локальные маркеры источников знания, убеждений, принципов, причинности, self-model и неизвестности служат только подсказками; запись идёт тем же capture-first путём через draft-атомы и дословные evidence-цитаты.
- **Capture-first разделение труда:** бот через выбранный live-LLM provider только *захватывает* — создаёт **черновые** атомы `status: draft`. Выверенный граф (промоушн `draft → stable`, связи внутри и между 01-04, реальные противоречия, дедуп, MOC, русские имена) строит сильная модель вручную через скилл `reconcista`; `05_Общее`, настроение, психометрику и soft skills — отдельный скилл `depersonalization`.
- **LLM только анализирует — запись в БД делает код:** в режиме `process` модель возвращает лишь `worldview_observations` (`area/category/theme/type/name/summary/quote/confidence`), `reaction`, опциональный `user_delta` и слабый `mask_frequency_draft`. Она НЕ придумывает slug, НЕ решает create-vs-update, НЕ пишет raw и НЕ правит curated `mask_frequencies.json`. Всё это детерминированно в коде: raw пишется дословно, slug выводится из имени (транслит), дубли ловит дедуп по всему worldview-графу.
- **Свободные заметки (`/ucho`):** verbatim в `00_raw/notes/<дата>.md` — обязательный первый шаг. Если запись заметки не прошла, LLM-разбор не стартует; если заметка сохранена, но LLM/граф-комментарий сорвался, бот молчит. Пользователю уходит только комментарий-реакция Иуды, без статуса «заметка сохранена» и счётчиков.
- **Дедупликация (live, лёгкая, в коде):** для каждого `worldview_observation` код прогоняет `worldview.resolve_slug` (по имени → по выведенному slug → aliases) + Jaccard-overlap на биграммах summary (0.7) против существующих атомов во всех областях. Совпадение → дописать evidence + alias, не создавать дубль. Глубокий дедуп/слияние — у strong-pass.
- **Портрет (`/about`):** отформатированный отдельным промптом текст «каким я тебя вижу» из `05_Общее/about.md`, плюс настроение из `01_Мироощущение/mood/mood.md` и психометрический профиль `05_Общее/profile.md`, если они уже собраны. Затем обычная сессия-обсуждение. `05_Общее/summary.md` хранит краткую сводку о пользователе и сжатые выводы по 01-04.
- **Очередь ответа во время генерации:** если человек досылает обычный текст, пока предыдущий ответ уже в LLM, бот склеивает новые сообщения через пустую строку в один `queued_answer` и потом обрабатывает его как продолжение старого вопроса одним LLM-запросом. `/echo <текст>` в busy-состоянии добавляется туда же. Другие команды отвечают `Ещё думаю.` и не закрывают сессию. `/start` удаляет только очередь, если она ещё не ушла в LLM, и не прерывает текущую генерацию.
- **Постоянство:** активная сессия, сквозной счётчик Q-номеров и ссылка на недоработанный ответ (`pending_answer_event_id`) переживают рестарт контейнера (`_session.json`, `_state.json`, atomic writes). Офлайн-ответы доезжают из очереди Telegram при следующем старте polling.
- **Полный session raw-log:** кроме человекочитаемого `00_raw/qna/YYYY-MM-DD.md`, бот ведёт машинный `00_raw/sessions/<session_id>.jsonl` со всеми сообщениями активной сессии, ролями, датами Telegram, `message_id`, `q_num`, `area/category/theme` и лицом Иуды.
- **Self-check при старте:** механический (без LLM) прогон — MOC rebuild по областям 01-04, проверка битых межпапочных ссылок, симметрии `contradicts`, self-link, orphan atoms и дублей summary в `.psycho/startup-check.md`.
- **Продолжение прошлых разговоров (reply-resume):** reply (смахни сообщение) на любую реплику сессии оживляет её с того места. Источник поиска — `00_raw/sessions`; совместимые `qmap/questions/sessions` только читают этот event-log.
- **Ежедневный пинок:** APScheduler шлёт главный вопрос в `DAILY_HOUR` — **ровно один раз в день** на пользователя (общий дедуп по дате `last_daily_date` для cron и догона). Активная сессия и прошлые ответы НЕ блокируют (как `/ask`). Если бот лежал в час рассылки — дневной **досылается позже в тот же день** при старте (за прошлые дни — нет). В `DAILY_REMINDER_START` (по умолчанию `23:00`) бот собирает тех, кто не ответил именно на сегодняшний daily Q, выбирает одно случайное batch-время до `DAILY_REMINDER_END` (`01:00`) и шлёт короткое напоминание в произвольной маске; оно не создаёт новый Q и не меняет якорь ответа.

### Secondary

- **Obsidian-native рендеринг:** callouts `[!summary]/[!quote]/[!question]/[!contradiction]/[!source]`, aliases во frontmatter, block refs `[[00_raw/qna/<date>#^Q<n>]]` (точная навигация в raw).
- **Психометрический профиль (строит depersonalization):** `05_Общее/profile.md` — оценка личности по корпусу ответов: **OCEAN (Big Five)** баллами 0–100, **MBTI** (тип) и **DISC** (стиль). Делает сильная модель вручную по тексту, с явными оговорками о точности и evidence-цитатами.
- **Soft skills (строит depersonalization):** `05_Общее/softskills.md` — оценка рабочих и межличностных навыков по ответам: коммуникация, кооперация, мышление/креативность, самоорганизация/лидерство. Бот этот файл не создаёт и не подмешивает напрямую в live-промпт.
- **Граф настроений (строит depersonalization):** отдельный контур `01_Мироощущение/mood/` — события, feedback, `_mood_map.json`, baseline и нарратив `mood.md`.
- **Наглядный русский граф (строит reconcista):** у выверенных (`stable`) атомов имя файла = `slug` = русский заголовок; links между атомами path-based и работают между областями 01-04. Узлы окрашены по области.
- **Area MOC:** `<01-04 область>/MOC.md` — оглавление области, атомы сгруппированы по category/type, пересобирается после изменений.
- **Защита ручных правок:** если файл концепта правился в Obsidian (или изменился после `git pull`), бот **не перезаписывает** — записывает warn в `.psycho/log.md` и сохраняет правки пользователя.
- **Операционный лог:** `.psycho/log.md` append-only, фиксирует drift-skip, dedup, sanitize, LLM-фолбэки.
- **Спиннер:** анимированный dice + текстовый placeholder во время ожидания LLM. Placeholder удаляется по готовности, dice остаётся как маркер «здесь думал».

### Out of scope

- Открытый/публичный multi-user, регистрация, MFA, 152-ФЗ-режим — нет. Multi-user есть, но **только для нескольких доверенных** по whitelist (владелец добавляет вручную); у каждого изолированная база. Открытый доступ и связанный с ним hardening — MVP B.
- Веб-интерфейс / мобильное приложение — клиент только Telegram, чтение результата только в Obsidian.
- Локальный LLM-контур удалён из runtime. Исторический baseline оставлен только в документации; live-маршрут работает через OpenRouter при наличии `OPENROUTER_API_KEY`, иначе через AITunnel.
- Голосовые сообщения, картинки, файлы — не разбираются и не сохраняются; бот отвечает короткой миниатюрой «бедное ухо без глаз» и ждёт текст.
- Экспорт графа в другие форматы (Notion, Roam, JSON-API) — Obsidian-vault считается source of truth.
- Бэкапы по cron, мониторинг, дашборды — это обвязка MVP A, на PoC B не делаем.
- Миграция исторического vault в мировоззренческую схему — `scripts/migrate_worldview_schema.py` (`dry-run` по умолчанию, `--apply` для переноса). `00_raw` byte-stable и не переписывается.

### Edge cases

Как пользователь взаимодействует с системой через чат в пограничных и неоднозначных состояниях. Демо-сценарии для показа вынесены в `.docs/demo.md`.

#### Доступ (на каждом сообщении)

- **Чужой (не в whitelist)** — бот молчит, команды в меню ему не видны.
- **Доверенный гость** — полный базовый набор команд, своя изолированная база.
- **Владелец** — базовый набор + админ-команды.
- Любая команда, **кроме `/pebble`, `/regen`, `/like`, `/remask`, `/leta`**, закрывает активную сессию перед выполнением (отправил команду посреди разговора — оборвал текущую сессию; данные целы; перегенерация, избранное, смена маски и предупреждение удаления не обрывают разговор). Пока ответ уже в LLM или ждёт в `queued_answer`, все команды кроме `/start` и `/echo <текст>` получают `Ещё думаю.` и сессию не закрывают; `/start` безопасно удаляет только очередь и не прерывает текущую генерацию.

#### Основной цикл «вопрос → ответ → реакция»

1. Приходит главный вопрос (`/ask`, дневной в `DAILY_HOUR`, `/echo`).
2. Пользователь отвечает свободным текстом.
3. Бот пишет сырой Q&A и черновые концепты, затем роняет **реакцию-укол** от первого лица (не вопрос). Под реакцией нет кнопок и служебной подписи лица; вместо неё отдельной строкой идёт короткий курсивный P.S. выбранной маски. Перегенерация и избранное доступны reply-командами.
4. Сессия **остаётся открытой**: реакция — якорь следующего хода, ответ на неё снова разбирается. Уточняющих вопросов бот не задаёт принципиально.

#### Задать вопрос

- **`/ask`** без аргумента — выбор области inline-кнопками (01-04 + «на выбор бота»); дальше категория и тема выбираются программно.
- **`/ask <тема>`** (тема из списка 10) — вопрос внутри названной темы.
- **`/ask <свободный текст>`** — текст становится targeted-затравкой: код мапит его к ближайшей области/категории/теме или создаёт одноразовый `custom_theme`.
- **`/echo <вопрос>`** — пользователь сам формулирует вопрос; бот возвращает его с меткой «пользовательский» и **присваивает Q-номер**. LLM при этом не зовётся.

#### Свободный ввод

- **Ответ на текущий вопрос** — обычный текст → разбор и реакция.
- **Текст во время генерации** — добавляется в один отложенный ответ через пустую строку; бот отвечает `Ещё думаю.`.
- **Reply на сообщение** — оживляет сессию из `00_raw/sessions` и продолжает её с места.
- **Ответ после вечернего напоминания** — ложится под исходный daily-вопрос: reminder пишется в session-log как `kind=reminder`, но не становится новым вопросом.
- **`/ucho <текст>`** — свободная заметка: сохраняется verbatim в `00_raw/notes/` и прогоняется через тот же разбор (работает как ответ без заданного вопроса).

#### Портрет и сервис

- **`/about`** — текст «каким я тебя вижу» из портрета, затем обычная сессия-обсуждение (можно отвечать reply). Портрет пуст → честно «ещё не распробовал, поговори».
- **`/pebble`** — статичное «Больно.» от Иуды, в которого бросили камень; **не трогает сессию**.
- **`/help`** — подсказка по командам. **`/start`** — смыв: закрыть сессию и удалить отложенный queued-ответ, если он ещё не отдан в LLM; текущую генерацию не прерывает. **`/leta`** — опасная команда удаления рабочей базы текущего пользователя; без аргументов показывает предупреждение, подтверждение только точной фразой `/leta УДАЛИТЬ <telegram_user_id>`. Удаляется `users/<uid>/`, доступ и git history не чистятся.
- **`/regen [маска]`** reply на комментарий Иуды — отправить отдельную новую реплику в ещё не использованной противоположной маске или в явно указанной маске; не пишет отрицательную оценку и не трогает граф.
- **`/like`** reply на реплику Иуды — добавить её в избранное; не закрывает сессию.
- **`/remask`** reply на вопрос или комментарий Иуды — открыть меню всех лиц и сменить выбранную маску; если reply не на сообщение бота из session-log, бот отвечает отказом.

#### Админ (только владелец)

- **`/adduser <id>` / `/removeuser <id>` / `/users`** — управление доступом (без LLM). `/removeuser` убирает доступ, но папку данных оставляет.
- **Reply-действия над ответом Иуды** — `/regen [маска]` отправляет отдельный новый reply в сильно другой ещё не использованной или явно выбранной маске; `/like` добавляет ответ в избранное (`1` в feedback). Сохранение работает и для перегенерированных ответов. Вопросы не оцениваются и не перегенерируются. `/remask` показывает меню всех лиц и меняет маску уже отправленного вопроса/комментария без запуска нового анализа графа.

#### Настроение и лицо

- Вялый/ровный/скучающий ответ → бот **раскачивает** (подначка, насмешка).
- Яркий/самоуверенный/высокомерный → **осекает** (сомнение, давление на больное, холод).
- Хрупкий/придавленный/тревожный → **поддерживает** (ласка, вера, вселение уверенности, клятва).
- Настроение копится по сессии (день может тянуться) — вес у последнего сообщения, фон берётся из портрета. Со временем depersonalization подстраивает выбор лица под то, что человеку терпимо, учитывая явные оценки перегенерации и избранные ответы.

#### Сбои и нестандартный ввод

- **Занят / очередь:** один активный LLM-вызов на пользователя + cooldown. Второй обычный текст во время обработки склеивается в `queued_answer`; `/echo <текст>` добавляется туда же; остальные команды отвечают `Ещё думаю.`. `/start` удаляет очередь, если она ещё не стартовала, и не прерывает уже начатый LLM-вызов.
- **LLM упал / таймаут / не-JSON:** нейтральная фраза пользователю, stacktrace в лог, граф не портится; неизвестные area/category/theme коэрсятся в валидную цель или игнорируются.
- **Рестарт посреди ответа:** двухфазный коммит (`pending_answer_event_id` на диск до LLM) — на старте ответ дожимается из `00_raw/sessions`, задвоения нет.
- **Бот лежал — офлайн-бэклог:** входящий текст одного пользователя **склеивается в один ответ** (один итоговый укол, а не на каждое сообщение); команды переигрываются по одной.
- **Пустой / слишком длинный ввод:** пусто после очистки → просьба переформулировать (LLM не зовётся); длинное — обрезается с предупреждением.
- **Reply на старую сессию:** поиск идёт по `00_raw/sessions`; если событие не найдено, фолбэк-цепочка реконструирует вопрос из текста сообщения или трактует ввод как заметку.
- **Несколько пользователей:** полная изоляция данных (`users/<id>/`), один git-коммит = данные одного пользователя.
- **Старая inline-кнопка устарела:** если action token уже вытеснен из `05_Общее/face_actions.json`, бот отвечает коротким alert и ничего не пишет.
- **Дедуп на лету:** повтор концепта (по имени / Jaccard) дописывает evidence, новый узел не плодит.
- **Write-barrier:** выверенные `stable`-концепты (русский slug) бот физически не перезаписывает; новые цитаты к ним доносит reconcista. Бот не удаляет отдельные файлы графа автоматически; исключение — явная `/leta`, которая стирает всю рабочую базу текущего пользователя после подтверждения.

## quality

**Нефункциональные требования:**

- **Приватность:** live-тексты уходят во внешний OpenRouter или AITunnel API; ключи хранятся только в `.env` как `OPENROUTER_API_KEY`/`AITUNNEL_API_KEY`. Это уже не офлайн-контур, поэтому бот остаётся whitelist-only для доверенных пользователей.
- **Сохранность данных:** ручные правки в Obsidian не должны теряться при последующей автозаписи. Закрыто manifest+mtime drift detection.
- **Целостность под падением:** kill контейнера в момент записи не должен оставить битый JSON / полу-записанный концепт. Закрыто atomic writes (tmp + os.replace).
- **Откат на ошибку:** падение посреди `_apply_processed` должно откатывать vault к pre-state. Закрыто `git_wrap` транзакцией (pre/post commit, на исключение `git reset --hard`).
- **Latency:** зависит от выбранного provider и его лимитов; таймаут одного вызова `LLM_TIMEOUT`, OpenRouter/AITunnel имеют fallback-маршруты.
- **MVP A readiness:** главный путь закрыт pytest smoke-тестами без живого Telegram/LLM provider, зависимости зафиксированы lock-файлом, generated artifacts выведены из индекса. До альфы остаются бэкапы/restore drill, runbook/deploy, структурные логи и минимальная операционная панель.

**Метрики успеха (для PoC B):**

- За неделю владелец отвечает ≥10 вопросов без падения бота.
- Граф в Obsidian-Graph-View визуально читается: видны кластеры по областям 01-04; связи между атомами появляются после `reconcista`, не live-ботом.
- Ни один автоматический коммит бота в vault не разрушил ручную правку.
- Минимум один conflict-callout после ручного прогона `reconcista` записан и помог осознать противоречие в собственной позиции.

## rules

### Hard

- LLM-вызовы идут только через внешний OpenAI-compatible live-маршрут: OpenRouter preferred, AITunnel fallback; локального LLM runtime в проекте больше нет.
- Все команды и подсказки видны только владельцу.
- Vault не теряет ручных правок: drift → skip + log, не overwrite.

### Firm

- Концепты — атомарные. Один концепт = одна идея, не лонгрид.
- **Slug выводит код, не LLM.** Черновик: ascii-slug из имени через `slugify` (транслит). Выверенный (`stable`, reconcista): имя файла = slug = русский заголовок (наглядный граф), старый ascii-slug — в `aliases` (якорь дедупа + back-compat ссылок).
- Парсер обратимо понимает старый формат (H2-секции) — на случай старых концептов и ручного редактирования.

### Soft

- В Telegram-сообщениях минимум разметки. Заголовок коротко, тело — `<code>`.
- LLM-фолбэки (неизвестная область/категория/тема, кривой type/status) — логируем, не падаем.

## accept

PoC B считается пройденным, когда:

- Бот запускается одной командой (`docker compose up -d`) и не падает в течение недели.
- Все 4 области реально работают: `/ask <area>` или случайный выбор дают осмысленный вопрос, ответ корректно разбирается LLM, атомы появляются в `<area>/atoms/`.
- Открытие MOC-ноды `<ТЕМА>.md` в Obsidian показывает все концепты темы, сгруппированные по type.
- Open Graph View с фильтром по `*/atoms` визуально читается — узлы и связи на месте.
- Конфликт-callout (`[!contradiction]`) реально записан хотя бы раз после `reconcista` и виден в Reader-режиме Obsidian.
- `.psycho/log.md` существует, наполняется и не пустой.
- При ручной правке `.md` в Obsidian → следующий ответ не перезатёр правку.
