# Weekly Review — ревизия базы знаний Psycho

Ты — сильный аналитический агент. Локальная модель бота (Qwen 14B) хорошо ведёт
диалог, но плохо структурирует. Твоя работа — раз в неделю прочитать, что
накопилось, и выдать человеку **наглядный обзор** + список проблем базы.

## Жёсткие границы (НЕ нарушать)

- **v1 — read-only для существующего вольта.** Ты читаешь `raw/`, `concepts/`,
  `profile/`, `.psycho/log.md`. Ты **НЕ редактируешь** их.
- **Пишешь ТОЛЬКО** в `<vault>/digests/`. Один файл за прогон.
- **Никогда** не трогаешь `_session.json`, `_state.json`, `.psycho/manifest.json`,
  `concepts/`, `profile/`, `raw/`, `_index.md`.
- Проблемы базы (дубли, слабые концепты) ты **перечисляешь как рекомендации**,
  но НЕ применяешь. Их применение — отдельный шаг v2 (см. конец файла).

## Где вольт

Путь к вольту берётся из `VAULT_HOST_PATH` в `<project>/.env`
(по умолчанию `C:/Users/eva/YandexDisk/Obsidian/Psycho`). Если не уверен —
спроси пользователя или прочитай `.env`-пример. Дальше `<vault>` = этот путь.

## Структура вольта (что читать)

```
<vault>/
├─ raw/<YYYY-MM-DD>.md        # сырые Q&A дня. Записи: «## Q<N> · HH:MM · <domain>», далее **Q:** / **A:**
├─ concepts/<domain>/<slug>.md  # узлы графа (кроме _moc.md и _*.md)
├─ profile/<domain>.md        # текущая свалка цитат по домену (контекст, не правим)
├─ .psycho/log.md             # операционный лог бота (warnings за неделю)
└─ digests/                   # СЮДА пишем (создать если нет)
```

10 доменов: `ethics, aesthetics, politics, everyday, relationships, identity,
mortality, nationality, knowledge, work`.

Формат концепт-ноты (frontmatter + тело):
```markdown
---
type: principle|value|preference|belief|claim
domain: <один из 10>
slug: <kebab-ascii>
status: stable|tentative|contested
supports: ["[[slug]]"]
contradicts: ["[[slug]]"]
derived_from: ["[[slug]]"]
related: ["[[slug]]"]
---
# Имя концепта
<summary 1-3 предложения>
## Подтверждения
- <дата> — «<цитата>» — из [[raw/<date>|Q<N>]]
## Открытые вопросы
- vs [[slug]]: <probe>
```

## Процедура

### Шаг 1 — окно
1. Определи текущую ISO-неделю в формате `YYYY-WNN` (например `2026-W21`).
   Bash: `date +%G-W%V`.
2. Диапазон — последние 7 дней (включая сегодня).
3. Если `<vault>/digests/<YYYY-WNN>.md` уже есть — спроси пользователя:
   перезаписать или пропустить. Не плоди `-1`, `-copy`.

### Шаг 2 — собрать материал
- **Raw недели:** `Glob <vault>/raw/*.md`, отбери файлы с датой в окне, прочитай.
  Извлеки все Q&A (заголовок `## Q<N> · HH:MM · <domain>`, поля `**Q:**` / `**A:**`).
- **Весь граф:** `Glob <vault>/concepts/*/*.md`, **исключи** имена на `_`
  (`_moc.md` и т.п.). Прочитай каждый: frontmatter + summary + evidence + open_questions.
- **Профили:** прочитай `<vault>/profile/*.md` для контекста (НЕ для правки).
- **Лог:** `<vault>/.psycho/log.md` — последние записи (truncations, drift, fallback).

### Шаг 3 — анализ (твоё суждение)

Раздели наблюдения на:

1. **Что проступило за неделю** — какие домены задеты, какие позиции человек
   проявил. Конкретно, со ссылками на концепты.
2. **Напряжения** — реальные противоречия. Источник: frontmatter `contradicts`
   + `## Открытые вопросы`. **Фильтр:** если в `raw` видно, что пользователь
   явно отверг рамку противоречия (фразы «не противоречие», «никак не объясняю»,
   «не вижу конфликта») — пометь напряжение как **снятое пользователем**, не
   подавай как актуальное.
3. **Здоровье базы** (это и есть proposal для v2, НЕ применяй):
   - **Дубли** — концепты с почти одинаковым summary или именем. Приведи
     пары/тройки slug-ов. Пример из текущей базы: `otkrytie-sushchestvovaniya-cherez-podarki`
     ≈ `podarki-kak-otkrytie-sushchestvovaniya` ≈ `napominanie-svoego-sushchestvovaniya`.
   - **Слабые** — 1 evidence, ноль связей, давно `tentative`.
   - **Осиротевшие** — ноль связей вообще.
   - **Свалки** — `profile/<domain>.md` где >5 несгруппированных цитат подряд.
4. **Недостающие связи** — концепты, которые стоило бы связать
   (related/supports/contradicts), но они не связаны.

### Шаг 4 — сгенерировать digest
- Файл: `<vault>/digests/<YYYY-WNN>.md`.
- Структура — строго по `digest-template.md` (рядом с этим SKILL.md).
- Obsidian-native обязательно:
  - `[[slug|Имя]]` — все упоминания концептов как кликабельные ссылки.
    Slug бери **точно** как имя файла концепта (без `.md`), иначе ссылка битая.
  - Callouts: `> [!summary]`, `> [!warning]`, `> [!todo]`, `> [!question]`.
  - Frontmatter с `type: digest`, `week`, `created`, `domains_touched`.
- Тон: ясный, по делу, без воды. Это обзор для самого человека о себе.

### Шаг 5 — навигация
- Создай/дополни `<vault>/digests/_index.md`: список ссылок на все digest-ы
  (новый сверху). НЕ трогай корневой `<vault>/_index.md`.

### Шаг 6 — отчёт
- Короткое summary пользователю в чат + путь к digest.
- Спроси: «Применить рекомендации из „Здоровья базы“?» — это вход в v2.

## Проверка перед завершением

- `git -C <vault> status --short` → изменения **только** в `digests/`.
  Если затронуто что-то ещё — откати свои правки, ты вышел за границы.
- Все `[[slug]]` в digest соответствуют реальным файлам `concepts/*/<slug>.md`.

## v2 — применение рекомендаций (НЕ делать без отдельного запроса)

Когда пользователь захочет применить findings, мутации идут через готовую
инфраструктуру проекта (всё внутри Docker, правило CLAUDE.md):
- `bot/vault.py::git_wrap("op_name")` — атомарная транзакция с git-откатом.
- `bot/atomic.py::atomic_write_text` — атомарная запись файла.
- `bot/moc.py::rebuild_domain_moc(domain)` — перестроить MOC домена.
- Паттерн dry-run + `--apply` через JSON-proposal — см. `scripts/migrate_domains.py`.

Операции v2: merge дублей (+ переписать wikilinks + удалить файлы), переписать
`profile/<domain>.md` в живой обзор, rebuild MOC по смысловым кластерам,
авто cross-links. Каждая — отдельный proposal → review → apply.
