# Weekly Review — сборка базы знаний Psycho

Ты — сильная модель. Архитектура проекта построена на разделении труда:

- **Бот (Qwen 14B, live)** только *захватывает*: ведёт диалог, пишет `raw/`,
  создаёт **черновые** концепты `status: draft` — БЕЗ связей и БЕЗ противоречий.
  Он намеренно не строит граф: он в этом слаб (плодит дубли и ложные конфликты).
- **Ты (раз в неделю)** *строишь* выверенный граф из черновиков и сырья:
  промоушн `draft → stable`, дедуп/слияние, связи, реальные противоречия,
  переписывание `profile/`, осмысленный MOC, и наглядный digest.

## Что тебе можно менять, а что нельзя

**Можно (это твоя зона):**
- `concepts/<domain>/<slug>.md` — промоушн, слияние, связи, контр-callouts, статусы.
- `profile/<domain>.md` — переписывать из свалки цитат в связный обзор.
- `concepts/<domain>/<DOMAIN>.md` (MOC-нода, имя = тема заглавными) — пересобирать.
- `digests/<YYYY-WNN>.md` + `digests/_index.md` — создавать.

**Нельзя (территория бота / источники истины):**
- `raw/**` — сырой лог, неприкосновенен (источник правды для evidence).
- `_session.json`, `_state.json`, `.psycho/manifest.json`, `_index.md` — служебное бота.

**Безопасность:** вольт — git-репозиторий. **Перед любыми правками** сделай
checkpoint-коммит, **после** — финальный коммит (см. Фазу 2). Бот не перезатрёт
выверенные (`stable`) концепты, потому что у них **русский `slug`**, а
`safe_slug` бота принимает только ASCII → запись отклоняется (см. «Наглядный
граф»). ⚠️ Drift-detection на это НЕ полагается: manifest бота ключуется по пути,
а переименованный (русский) файл ему неизвестен → `check_drift` вернёт `False`.
Реальный барьер — русский slug, а не drift.

## Где вольт (multi-user)

Путь к вольту — из `VAULT_HOST_PATH` в `<project>/.env` (дефолт
`C:/Users/eva/YandexDisk/Obsidian/Psycho`).

**Бот multi-user:** данные каждого пользователя — в `<vault>/users/<user_id>/`.
Дальше `<base>` = корень КОНКРЕТНОГО пользователя (`<vault>/users/<user_id>/`).
- По умолчанию работай с **владельцем** (id из `OWNER_TELEGRAM_ID` в `.env`).
- Для ревизии другого пользователя — возьми его id, работай в его `users/<id>/`.
  **Никогда не смешивай данные разных пользователей.**
- `.psycho/` (manifest, log, users.json) и `.git/` — ГЛОБАЛЬНЫЕ на корне вольта;
  не трогаешь (log — только чтение).

## Структура и форматы

```
<vault>/users/<user_id>/         # = <base>, корень одного пользователя
├─ raw/<YYYY-MM-DD>.md           # Q&A дня: «## Q<N> · HH:MM · <domain>», **Q:** / **A:**, ^Q<N>
├─ concepts/<domain>/<slug>.md   # узлы (кроме _*.md). Бот пишет draft, ты доводишь до stable
├─ profile/<domain>.md           # пока свалка цитат — переписываешь в обзор
└─ digests/                      # твои обзоры
<vault>/.psycho/log.md           # глобальный лог (общий, только чтение)
```
Все пути ниже (`raw/`, `concepts/`, …) — **относительно `<base>`** пользователя.

10 доменов: `ethics, aesthetics, politics, everyday, relationships, identity,
mortality, nationality, knowledge, work`.

**Черновик бота** (что ты получаешь на вход): `slug: <kebab-ascii>`,
`status: draft`, без связей, тело часто в легаси-формате `## Подтверждения` /
`## Открытые вопросы`. Это сырьё для доводки.

**Выверенный концепт** (что ты пишешь — `status: stable`):
```markdown
---
type: principle|value|preference|belief|claim
domain: <один из 10>
slug: <Русское имя файла>          # = имени файла (раздел «Наглядный граф»)
status: stable
supports: ["[[Русское имя]]"]
contradicts: ["[[Русское имя]]"]
derived_from: ["[[Русское имя]]"]
related: ["[[Русское имя]]"]
aliases: [старый ascii-slug, прежние формулировки]   # ascii — якорь дедупа
tags: [<DOMAIN-CAPS>, <сквозная-тема>, …]            # доменный тег = имя папки CAPS; из реестра _tags.md
---
# Русское имя

> [!summary] TL;DR
> Законченное предложение в одну строку (его берёт MOC).

> [!quote]
> цитата из raw
> — из [[raw/<date>#^Q<N>]] · <дата время>

> [!question]
> открытый вопрос / probe
```
`status: draft` = создан ботом, не выверен. Твоя работа — довести до `stable`
(`tentative`/`contested` — твои промежуточные пометки, их тоже доводи).
Контр-противоречие — callout `> [!contradiction] vs [[Русское имя]]` в ОБА
концепта плюс симметричная связь `contradicts` во frontmatter обоих.

## Наглядный граф: русское имя = русский slug

Цель: узлы графа Obsidian подписаны **по-русски**. На графе узел = **имя файла**,
ребро = `[[ссылка]]`. Значит у выверенного (`stable`) концепта на русском **всё
сразу**: имя файла, `slug` и все ссылки на него. Это принцип kepano — *имя файла
это заголовок, ссылки ведут по имени, а не по id*.

**Как именовать (slug == имя файла == русский заголовок):**
- Имя файла = `slug` = человекочитаемое русское имя из `# Имя`:
  `Неприятие вялости в людях.md`, во frontmatter `slug: Неприятие вялости в людях`.
- Все ссылки — на русское имя: `[[Неприятие вялости в людях]]` (frontmatter
  `related`, тело, profile, MOC, digest). Резолвятся по имени файла → русский
  узел, **без ascii-ghost-нод**.
- **Старые ascii-slug-и кладём в `aliases`** — это якорь для дедупа: бот (Qwen)
  предлагает ascii-форму, `resolve_slug` находит концепт по alias и не плодит
  дубль. На графе `aliases` не видны.

**Бот НЕ перезапишет выверенный концепт (важно, проверено):**
- `safe_slug` бота принимает только ASCII → русский `slug` он привести не может,
  поэтому `append_evidence` / `add_alias` / `save_concept` по русскому slug
  возвращают `None`. **Русский slug = естественный write-barrier**: `stable` —
  территория weekly-review. Новые цитаты копятся в `raw/` и подхватываются на
  следующем прогоне (Фаза 0 читает `raw/`).
- ⚠️ На drift-protection НЕ полагайся для переименованных файлов: manifest бота
  ключуется по пути, новый русский путь ему неизвестен → `check_drift` вернёт
  `False`. Защищает именно **русский slug**, а не drift.

**Имя файла / slug — санитайзинг (Windows-safe):**
- Убери символы `< > : " / \ | ? *`, схлопни пробелы, без точки/пробела в конце.
- Не reserved-имя Windows (`con`, `nul`, `prn`, `aux`, …). Длина ≤ ~80 символов.
- Сохраняй регистр и пробелы (наглядность). При коллизии добавь ` 2`, ` 3`.
- Переименование — только через git: `git -C <vault> mv "<старое>.md" "<Русское имя>.md"`.

## Теги: сквозные темы + домен

Второе измерение графа помимо `[[ссылок]]`: при «Show tags» каждый тег —
**узел**, концепты с общим тегом стягиваются в кластер. Уникальная польза —
**сквозные мотивы через РАЗНЫЕ домены** (то, чего не дают папки и MOC). Теги —
НЕ замена `related`: смысловая связь концепт↔концепт остаётся `[[ссылкой]]`,
тег — общий мотив-хаб.

- Теги — в `tags:` frontmatter (без `#`), **без пробелов** (Obsidian не
  разрешает; слова через дефис).
- **Доменный тег = имя папки домена ЗАГЛАВНЫМИ** (англ.): `domain: aesthetics`
  → `AESTHETICS`. На графе это **узел-хаб домена**, заменяющий MOC-файл
  `<DOMAIN>.md` (тот скрыт с графа фильтром, см. ниже). Ровно один на концепт,
  совпадает с `domain`.
- **Сквозные темы — русские** (`яркость`, `присутствие`, …), 1–3 на концепт.
  Больше — не надо, иначе hairball.
- **Закрытый реестр — `<base>/_tags.md`.** Новый тег появляется СНАЧАЛА там
  (осознанно), потом в концептах. Лежит вне `concepts/` → в граф концептов не
  попадает (источник правды для Фазы 3 LINT).
- Домены (10): ETHICS, AESTHETICS, POLITICS, EVERYDAY, RELATIONSHIPS, IDENTITY,
  MORTALITY, KNOWLEDGE, WORK, NATIONALITY.

**Настройки графа (`.obsidian/graph.json`, локальный UI):**
- `showTags: true`, `showOrphans: false` (без узлов-сирот), `hideUnresolved: true`.
- **MOC-файлы (`<DOMAIN>.md`) скрываем с графа** — их роль хаба берёт доменный
  тег. ⚠️ Property-фильтр `["type":"moc"]` в графе НЕ работает (проверено) —
  исключай **по имени файла**: добавь в `search`
  `-file:ETHICS -file:AESTHETICS -file:POLITICS -file:EVERYDAY -file:RELATIONSHIPS
  -file:IDENTITY -file:MORTALITY -file:KNOWLEDGE -file:WORK -file:NATIONALITY`
  (концепты русские → под эти ascii-токены не попадают).
- **Цветовые группы по доменам:** один `colorGroups`-элемент на подпапку —
  `{ "query": "path:concepts/<домен>", "color": {...} }`, свой цвет → узлы
  концептов окрашены по домену. **Узлы-теги — серые:** последняя группа в списке
  с запросом `tag:#... OR ...` и серым цветом (домены выше → выигрывают для
  концептов; теги попадают только в серую группу). Приоритет — у верхней
  совпавшей группы.
- **Per-user `<base>/.obsidian/graph.json` сидится автоматически** ботом из
  шаблона `bot/assets/graph.json` (`vault._ensure_user_graph_settings`, при
  создании папки юзера; существующий НЕ перезаписывает). Шаблон = эталон:
  фильтр `path:concepts` + MOC-исключения + цвета доменов + серые теги. Менять
  дефолт для НОВЫХ юзеров — правь `bot/assets/graph.json`.
- Корневой вольт (база владельца) настраивается отдельно: `search:
  "path:users/<OWNER_ID>/concepts <MOC-исключения>"` (фильтр абсолютный).
- ⚠️ Obsidian **перезаписывает `graph.json`** при правках графа в GUI. После
  внешних изменений — Reload (Ctrl+R); не меняй настройки графа в GUI, иначе
  фильтр/цвета затрутся.

## Процедура: две фазы (proposal → apply)

### Фаза 0 — окно и сбор
1. ISO-неделя: `date +%G-W%V` → `YYYY-WNN`. Окно — последние 7 дней.
2. Прочитай: `raw/` за неделю; ВСЕ `concepts/*/*.md` (кроме `_*.md`) с frontmatter+телом, выдели `status: draft`; `profile/*.md`; `.psycho/log.md`.

### Фаза 1 — PROPOSAL (ничего не пишем в граф)
Проанализируй и составь **план сборки**. Покажи его пользователю в чате (кратко) и запиши в `<vault>/.psycho/weekly-proposal-<YYYY-WNN>.md`. План включает:

1. **Промоушн черновиков.** Для каждого `draft` (а также не доведённых
   `tentative`/`contested`) концепта реши:
   - **merge** в существующий (дубль/почти-дубль) — указать в какой, перенести evidence + alias;
   - **promote** как новый `stable` (уточнить name/summary/type, выдать русский slug=имя файла);
   - **drop** как шум (объяснить почему).
2. **Связи.** Какие `supports / derived_from / related` проставить между концептами (с обоснованием в одну строку).
3. **Реальные противоречия.** Только настоящие (не формальное противопоставление слов). **Фильтр:** если в `raw` пользователь явно отверг рамку («не противоречие», «никак не объясняю») — НЕ ставить, отметить как снятое.
4. **Переписать profile/<domain>.md** — для доменов, где profile = свалка цитат: в связный обзор с кластерами и ссылками `[[Русское имя]]`.
5. **MOC** — какие домены пересобрать по смыслу.
6. **Новые цитаты к выверенным.** Пройди `raw/` недели: реплики, поддерживающие
   уже существующий `stable`-концепт, добавь к нему как `[!quote]`. Бот этого не
   делает — `stable` защищён русским slug, поэтому evidence доносишь только ты.
7. **Теги.** Каждому концепту — доменный тег `<DOMAIN-CAPS>` (= имя папки, напр.
   `AESTHETICS`) + 1–3 сквозные русские темы из реестра `_tags.md` (раздел
   «Теги»). Если мотив повторяется, а тега нет — предложи новый тег в реестр с
   обоснованием (часть плана, выносится на «да»).

Спроси: **«Применить план? да / нет / правки»**. Без «да» — стоп (proposal остаётся для просмотра в Obsidian).

### Фаза 2 — APPLY (по подтверждению)
1. **Checkpoint:** `git -C <vault> add -A && git -C <vault> commit -m "weekly-review <YYYY-WNN>: before apply" --allow-empty`.
2. Применяй план по порядку:
   - merge: перенести evidence/aliases в целевой концепт, обновить summary, удалить лишний файл, **переписать все `[[old]]` → `[[Русское имя survivor-а]]`** в других концептах и profile. **Survivor переименовать в русское имя**, его старый ascii-slug добавить в `aliases`.
   - promote: выставить `status: stable`, причесать summary/name. **Имя файла = `slug` = русское человекочитаемое имя** (раздел «Наглядный граф»): `git -C <vault> mv "<ascii>.md" "<Русское имя>.md"`, во frontmatter `slug: <Русское имя>`, старый ascii-slug → `aliases` (якорь дедупа).
   - связи: добавить симметрично во frontmatter обоих (`supports` A→B и B→A; `derived_from` A→B и `related` B→A). **Цель ссылки = русское имя файла**: `[[Русское имя]]`.
   - противоречия: `contradicts` в обоих + `> [!contradiction] vs [[Русское имя]]` callout в обоих.
   - evidence: добавь новые `[!quote]` из `raw/` недели к подходящим существующим `stable`-концептам (бот их не пишет — барьер русского slug).
   - теги: проставь `tags:` (доменный `<DOMAIN-CAPS>` + сквозные из реестра) в stable-концепты; обнови/создай реестр `<base>/_tags.md`; один раз — графовые настройки (`showTags`/`showOrphans`/`hideUnresolved`, исключение MOC по `-file:<DOMAIN>`, цветовые группы по подпапкам) в корневом и per-user `<base>/.obsidian/graph.json` (раздел «Теги»).
   - profile: перезаписать `profile/<domain>.md` — связный обзор по-русски, ссылки `[[Русское имя]]`.
   - MOC: пересобери ботом — он пишет `[[<slug>|<имя>]]` = `[[Русское имя|Русское имя]]` (slug теперь русский), что и нужно. **Не пиши MOC-ноду `<DOMAIN>.md` руками: бот перестраивает все MOC на каждом старте (`startup_selfcheck`) и затрёт ручную правку.** Команда (НЕ `docker compose run` — он поднимет ollama):
     ```bash
     # <repo> = корень проекта (где docker-compose.yml), <vault> = путь вольта,
     # <user_id> = пользователь, для которого собираем граф.
     # Нет образа psycho-bot? собери: docker compose -f <repo>/docker-compose.yml build bot
     MSYS_NO_PATHCONV=1 docker run --rm --entrypoint python \
       -v "<repo>":/app -v "<vault>":/vault -w /app --env-file <repo>/.env \
       psycho-bot:latest -c "from bot import userctx, moc; userctx.set_user(<user_id>); moc.rebuild_domain_moc('<domain>')"
     ```
     Bot берёт summary как **первую строку** `[!summary]` (обрезает до 200 симв.) — поэтому первая строка TL;DR должна быть законченным предложением (см. «Наглядный граф»).
3. **Digest:** создай `digests/<YYYY-WNN>.md` по `digest-template.md`.
4. **Навигация:** дополни `digests/_index.md` (новый сверху).
5. **Финальный коммит:** `git -C <vault> add -A && git -C <vault> commit -m "weekly-review <YYYY-WNN>: applied"`.

### Фаза 3 — LINT (проверка целостности, действие)
После APPLY (и по отдельному запросу «прогон lint») пройди базу. Механическое —
**чини**, по остальному — отчёт в чат. Проверки:
- **Битые ссылки:** каждая цель `[[Русское имя]]` совпадает с именем
  существующего файла концепта. Битые после merge — почини.
- **Инвариант русских имён** (для `stable`): имя файла = `slug` = русское; есть
  хотя бы один ascii-alias (якорь дедупа бота). Нарушение — почини.
- **Остаточные `status: draft`** среди обработанных — не должно быть (stable / merged / dropped).
- **Симметрия `contradicts`:** `[!contradiction]` обязан содержать `[[wikilink]]`
  и иметь парный `contradicts` в обоих концептах.
- **`[!quote]` без атрибуции; `[!todo]` в `status: stable`** — почини (todo только в draft).
- **Осиротевшие** (0 связей, 1 evidence, давно `tentative`) — в отчёт, не трогай.
- **Теги:** каждый тег концепта ∈ реестр `_tags.md`; доменный тег = `domain`
  заглавными (`aesthetics` → `AESTHETICS`); формат без пробелов; ровно 1 доменный
  + ≤3 сквозных. Тег вне реестра — внести в реестр осознанно или убрать.
- **Неприкосновенное не тронуто:** `raw/`, `_session.json`, `_state.json`,
  `.psycho/manifest.json`, `_index.md` (`git diff --name-only` checkpoint..HEAD).

## Закрытая схема callout-ов (из praxis)
Используем только эти типы; новые — расширять явно. Валидируется в Фазе 3.

| Тип | Назначение | Обязательно |
|---|---|---|
| `[!summary]` | TL;DR концепта/недели | — |
| `[!note]` | нейтральное примечание, «снято пользователем» | — |
| `[!question]` | открытый вопрос / probe | — |
| `[!warning]` | напряжение позиций | суть столкновения |
| `[!contradiction]` | реальное противоречие | `[[wikilink]]` на конфликтующий + парный `contradicts` |
| `[!quote]` | цитата из `raw/` | атрибуция `— из [[raw/<date>#^Q<N>]]` |
| `[!source]` | сессия/источник | ссылка на сессию |
| `[!todo]` | отложенное действие | **только** в `status: draft` |

Правила: один callout = одна мысль, не вкладывать друг в друга. `[!contradiction]`
ставится вместо тихого мержа — сохраняет нюансы обеих сторон. **Первая строка
`[!summary]` — законченное предложение** (≤200 симв.): MOC берёт именно её и
обрежет многострочный summary мид-фразой.

## Разовая миграция существующих концептов

При первом прогоне после апгрейда (под checkpoint-коммитом Фазы 2): пройди все
`concepts/*/*.md`. Для каждого выверяемого (после promote/merge) — переименуй в
русское имя (раздел «Наглядный граф»), выстави `slug: <Русское имя>`, старый
ascii-slug добавь в `aliases`. **Перепиши все ссылки `[[old]]` → `[[Русское имя]]`**
во всех концептах, profile, digest. У старых нот тело в легаси-формате
`## Подтверждения` — при ре-сейве приведи к callout-формату (`[!summary]`,
`[!quote]` с атрибуцией). Свежие `draft` от бота не трогай — переименуешь и
переслагуешь при их промоушене.

## Digest

`digests/<YYYY-WNN>.md` — наглядный обзор для человека, по `digest-template.md`
(callouts, `[[wikilinks]]`, frontmatter `type: digest`). Секции: что проступило,
напряжения (с пометкой «снято пользователем» где надо), что собрано/слито за
прогон, вопросы-кандидаты на следующую неделю.

## Если запускают «только обзор, без сборки»

Если пользователь просит just digest — выполни Фазу 0 + сразу digest (как
read-only обзор), пропусти Фазы 1–2. По умолчанию же скилл **собирает граф**.
